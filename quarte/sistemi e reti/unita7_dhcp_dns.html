La Configurazione degli Host

01 / 01

📋 Due tipi di attività

  • Host configuration: configurare ogni host prima della messa in rete (indirizzo IP, subnet mask, gateway, DNS)
  • Host management: gestire e controllare gli host durante il funzionamento (monitoraggio anomalie, degrado prestazioni)

📡 Il protocollo BOOTP

  • Predecessore di DHCP — RFC 951
  • Usa UDP, porte 67 (server) e 68 (client)
  • Invia IP e parametri in un solo messaggio
  • Supporta più subnet tramite broadcast IP

🔄 Procedura BOOTP (2 fasi)

  • Fase 1: il client riceve l'indirizzo IP e il nome del file immagine da scaricare
  • Fase 2: il client usa TFTP (Trivial File Transfer Protocol) per scaricare il file dal server
💡 DHCP e SNMP (Simple Network Management Protocol) lavorano entrambi a livello Application per indipendenza dall'hardware e per sfruttare le funzionalità del livello Transport.
📌 DHCP non è una "rivoluzione" rispetto a BOOTP: ne è un miglioramento, aggiungendo l'assegnazione dinamica degli indirizzi IP.

Il Protocollo DHCP

01 / 07

📌 Cos'è il DHCP?

Dynamic Host Configuration Protocol — RFC 2131

Consente l'assegnazione automatica e dinamica dei parametri di rete agli host, senza intervento manuale dell'amministratore.

🌐 Parametri assegnati dal DHCP

  • Indirizzo IP — identificatore dell'host
  • Subnet mask — per conoscere il prefisso di rete
  • Default gateway — router verso Internet
  • DNS Server preferito
  • DNS Server alternativo

⚠️ APIPA — Fallback senza server

Automatic Private IP Addressing: se nessun DHCP Server risponde, Windows assegna automaticamente un indirizzo nell'intervallo:

169.254.0.1 — 169.254.255.254
Subnet mask: 255.255.0.0  (Classe B default)
Gateway:     0.0.0.0

APIPA è un fallback temporaneo: quando il server torna disponibile, rilascia l'indirizzo e ne richiede uno tramite DHCP.

⚠️ Con un indirizzo APIPA, l'host può comunicare solo con altri host APIPA della stessa LAN — non ha accesso a Internet.

Architettura Client/Server DHCP

02 / 07

🔧 Manuale

L'amministratore assegna un IP specifico e permanente a un host preciso nel database del server.

Usato per router, server, stampanti di rete — dispositivi che devono avere sempre lo stesso IP.

⚡ Automatica

Il server assegna un indirizzo IP permanente in modo automatico a ogni nuovo host.

L'IP rimane fisso per quell'host, ma non richiede configurazione manuale dell'amministratore.

🔄 Dinamica

Il server assegna un IP per un tempo limitato (tempo di lease) secondo la lease length policy.

Allo scadere, il client può richiedere il rinnovo o un nuovo indirizzo.

✅ Vantaggi della configurazione dinamica

  • Automazione: nessun intervento manuale
  • Gestione centralizzata: tutto dal server
  • Condivisione: più host del pool disponibile
  • Portabilità: un host può cambiare subnet
  • No conflitti: IP presi dal database, nessun duplicato
📡 Relay Agent: se il DHCP Server non è nella stessa subnet del client, un router funge da intermediario, inoltrandone le richieste attraverso più reti.
🔁 Backup: si usano 2 DHCP Server (75%/25% degli indirizzi) per fault-tolerance e bilanciamento del carico.

1. DISCOVER

Il client invia in broadcast (255.255.255.255) sulla porta UDP 67 per trovare un DHCP Server disponibile.

2. OFFER

Ogni DHCP Server risponde in broadcast con un'offerta: IP proposto, subnet mask, gateway e tempo di lease.

3. REQUEST

Il client sceglie una delle offerte e invia in broadcast DHCPREQUEST. Gli altri server rientrano nel pool.

4. ACK

Il server selezionato risponde con DHCPACK, confermando tutti i parametri di configurazione. Da ora il client usa l'IP.

🧠 DORA = Discovery · Offer · Request · Acknowledgement — l'acronimo della procedura di assegnazione dinamica DHCP.

Gli Stati del DHCP Client

05 / 07
🤖 Il DHCP usa una FSM (Finite State Machine) per descrivere il ciclo di vita del lease dal punto di vista del client.
INITIALIZE

Stato iniziale. Invia DHCPDISCOVER in broadcast.

SELECT

Raccoglie le offerte DHCPOFFER e ne sceglie una.

REQUEST

Invia DHCPREQUEST al server scelto.

BOUND

Usa l'IP assegnato. Imposta i timer T1 e T2.

RENEW

Al 50% del lease: richiede rinnovo con DHCPREQUEST unicast.

REBIND

All'87,5%: DHCPREQUEST in broadcast a qualsiasi server.

⏱️ Timer T1 e T2

  • T1 (renewal timer): al 50% del lease → stato RENEW, invia DHCPREQUEST unicast al server
  • T2 (rebinding timer): all'87,5% del lease → stato REBIND, broadcast a qualsiasi server

🔄 Rilascio indirizzo IP

WINDOWS
ipconfig /release
ipconfig /renew
LINUX
dhclient -r [interfaccia]
dhclient [interfaccia]
MACOS
sudo ipconfig set [interfaccia] NONE
sudo ipconfig set [interfaccia] DHCP
💡 Se il client invia DHCPRELEASE, l'indirizzo torna immediatamente nel pool disponibile del server.

Il DHCP per IPv6 — DHCPv6

06 / 07

🆕 Due modalità di autoconfigurazione IPv6

  • Stateless (RFC 4862): l'host si configura autonomamente senza server — adatta a reti domestiche
  • Stateful (DHCPv6 — RFC 8415): un server fornisce le informazioni all'host — controllo centralizzato nelle reti aziendali

🔌 Porte UDP diverse da IPv4

  • UDP 546 → DHCPv6 Client
  • UDP 547 → DHCPv6 Server

Non compatibile con client IPv4 (che usano porte 67/68).

🏷️ Identificatori univoci

  • DUID (DHCP Unique Identifier): identifica l'host — generato una sola volta dal SO e memorizzato in modo permanente
  • IAID (Identity Association Identifier): identifica ogni interfaccia di rete dell'host

📨 Modalità a 4 messaggi (Stateful)

SOLICIT → multicast ff02::1:2 per trovare server disponibili
ADVERTISE ← i server rispondono con l'offerta
REQUEST → il client sceglie e richiede conferma
REPLY ← il server conferma l'assegnazione

📩 Modalità a 2 messaggi (Stateless)

Se il client ha già un indirizzo e vuole solo altri parametri (es. lista DNS):

← Reply
🔄 Il rinnovo si fa con un messaggio Renew (≈ DHCPREQUEST di DHCPv4). Il rilascio con Release.

Il DNS — Domain Name System

01 / 04
📌 Il DNS consente di usare nomi al posto degli indirizzi IP. È un database distribuito e organizzato gerarchicamente — RFC 1034 e RFC 1035.

🏗️ I 3 componenti del DNS

  • Domain Name Space: struttura ad albero dei nomi di dominio
  • Name Server: processo che gestisce informazioni su zone del Name Space — è un authoritative Name Server per le sue zone
  • Resolver: applicazione client che ottiene informazioni dal Name Server (es. gethostbyname in Unix)

🌳 Tipi di dominio

  • Domini radice: dominio di primo livello — TLD (Top Level Domain): .it, .com, .org…
  • Domini intermedi: hanno sottodomini (es. school.it)
  • Domini foglia: senza sottodomini, contengono solo host (es. di.school.it)

🔤 Come si compone un nome FQDN

Il nome si percorre dalla foglia alla radice, separato da punti. Il punto finale (radice) è implicito.

www.ietf.org
     ↑    ↑    ↑
  host dominio TLD (radice implicita)
  • Singoli componenti: max 63 caratteri
  • Nome completo: max 255 caratteri
  • Nomi case-insensitive: .IT = .it

🔑 Gestione dei TLD

IANA gestisce i TLD a livello mondiale. Collabora con ICANN che coordina le attività dei root Name Server.

Il Name Server è in ascolto su porta 53 UDP (query) e 53 TCP (trasferimento zone).

Come Funziona il DNS

03 / 04

🔍 Query iterative vs ricorsive

  • Iterativa: il client chiede a un server DNS, che risponde con la migliore risposta nota o con il rimando al prossimo server
  • Ricorsiva: il client chiede a un server DNS di rispondere in modo completo. Il server può interrogare altri server per conto del client
💡 I resolver fanno query ricorsive; i Name Server solitamente query iterative.

🔁 Esempio: risoluzione di www.company.com

1. Resolver → Default DNS Server (query A per www.company.com)
2. Default DNS → Root Server
3. Root Server risponde con NS del dominio .com
4-5. Default DNS → NS di .com → ottiene NS di company.com
6-7. Default DNS → Authoritative NS di company.com → IP
8. Default DNS → Resolver (risposta con IP = 138.34.121.75)

🔄 Risoluzione inversa

Dato un indirizzo IP, trovare il nome corrispondente. Si usa il dominio speciale in-addr.arpa.

Per risolvere 198.45.30.165 → si cerca il RR PTR di:

165.30.45.198.in-addr.arpa.

L'indirizzo IP è scritto al contrario: nella gerarchia DNS, le parti più specifiche sono a destra (al contrario degli IP).

💾 Cache DNS

  • I Name Server memorizzano le risposte in cache
  • Una risposta da cache è non-authoritative
  • Il campo TTL indica quanto a lungo tenere l'informazione
  • DNS primario e secondario si sincronizzano via TCP 53

Problematiche di Sicurezza

01 / 01

🔴 Vulnerabilità DHCP

DHCP usa UDP e IP — intrinsecamente insicuri. Nessuna autenticazione prevista nelle specifiche originali.

  • DHCP Server non autorizzato: un server abusivo risponde con parametri falsi, bloccando gli host o configurandoli per azioni fraudolente
  • DHCP Client non autorizzato: esaurisce il pool IP con migliaia di richieste, bloccando nuovi accessi alla rete (DHCP starvation)
🛡️ Soluzione: controllo fisico dell'accesso alla rete + IPSec al livello Network.

🔵 Vulnerabilità DNS

  • Non è autenticato: la risposta potrebbe non venire dal DNS Server corretto (spoofing)
  • DNS Cache Poisoning: modifica dei dati nella cache del server inserendo TTL alto — l'utente viene reindirizzato a siti malevoli
  • Non offre meccanismi per proteggere l'integrità delle informazioni

🔐 DNSSEC

Domain Name System Security Extensions: estensione del DNS definita dall'IETF.

  • Usa chiavi crittografiche per autenticare i dati DNS a partire dalla root
  • Le chiavi della root sono gestite da ICANN
  • 5 indicatori: coerenza, integrità, velocità, disponibilità, robustezza

🖥️ nslookup è disponibile su Windows, Unix e Linux. Permette di interrogare il DNS per risolvere nomi, indirizzi IP e specifici Resource Record.

Modalità interattiva

Si digita nslookup senza argomenti. Permette più query successive.

C:\>nslookup

Si passa il nome direttamente come argomento:

server NOME — cambia il DNS server da usare

exit — esce dal programma

>server 194.119.192.34

Lab 10 — Configurazione Server DNS

Due LAN con server DNS pubblico. Si configurano Resource Record di tipo A Record:

Server DNS — scheda Services
www.aaa.it → A Record → 192.168.1.100
www.bbb.it → A Record → 192.168.2.100

Ogni host della rete usa questo DNS Server per risolvere i nomi dei server web delle due LAN.

🔍 Per verificare: aprire il Web Browser di un PC, digitare www.aaa.it e controllare che venga risolta la pagina web corretta.

📡 Le query DNS usano UDP 53. Il trasferimento di zone tra server primario e secondario usa TCP 53.

🏳️ Il campo flag (16 bit)

  • QR: 0=query, 1=risposta
  • Opcode: 0=Standard, 1=Inversa, 2=Stato server
  • AA: 1 → risposta authoritative
  • TC: 1 → messaggio troncato (oltre 512 byte)
  • RD: 1 → richiesta ricorsiva
  • RA: 1 → ricorsione disponibile

📊 Codici di ritorno (Rcode)

0 — Nessun errore
1 — Errore costruzione query
2 — Errore interno nel server
3 — Nome non esiste nel dominio
4 — Tipo query non supportato

Riepilogo — Unità 7

🟢 DHCP

  • Assegna automaticamente: IP, subnet mask, gateway, DNS
  • 3 modalità: manuale, automatica, dinamica (lease)
  • Procedura DORA: Discover → Offer → Request → Ack
  • FSM: INITIALIZE → SELECT → REQUEST → BOUND → RENEW → REBIND
  • Timer T1 (50%) e T2 (87,5%) per il rinnovo
  • Relay Agent per reti multi-subnet
  • APIPA: fallback 169.254.0.0/16 senza server

🟡 DHCPv6

  • Stateless (RFC 4862) o Stateful (RFC 8415)
  • Identificatori: DUID (host) e IAID (interfaccia)
  • Porte UDP 546 (client) e 547 (server)
  • 4 messaggi: Solicit → Advertise → Request → Reply

🔵 DNS

  • Database distribuito: Domain Name Space, Name Server, Resolver
  • Struttura gerarchica: radice → TLD → dominio → foglia
  • Resource Record: A, AAAA, CNAME, MX, NS, PTR, SOA, TXT
  • Query iterative (NS) e ricorsive (resolver)
  • Risoluzione inversa via dominio in-addr.arpa
  • Porta 53 UDP (query) e 53 TCP (trasferimento zone)

🔴 Sicurezza

  • DHCP: server non autorizzati, DHCP starvation → IPSec
  • DNS: spoofing, cache poisoning → DNSSEC
  • DNSSEC: autenticazione crittografica dalla root (ICANN)
